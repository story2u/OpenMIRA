# ADR-0009：AI 自动回复使用确定性策略门禁和独立投递账本

> 状态：accepted · 日期：2026-07-15

全局开关的默认值后续由 [ADR-0010](0010-enable-ai-reply-pipeline-by-default.md) 调整为开启；本 ADR 的
多层授权、确定性门禁和投递语义保持有效。

## 背景

现有非工作时间路径会在 pi Agent 风险分析前排队 AI 回复，且没有用户来源级授权和可靠投递状态。
模型输出可能受外部消息或网页提示词注入影响；Celery/provider 边界也无法天然提供 exactly-once。
自动联系外部人员属于高风险、难撤销动作，不能只靠 prompt、客户端开关或机会状态控制。

## 决策

- pi Agent 保持无外发工具，只产出结构化分析；自动发送资格由 Python 纯策略确定。
- 引入独立自动回复投递账本，使用用户级稳定幂等键、明确状态和最小安全元数据。
- 采用 at-most-once 语义：发送前持久化 `sending`；未知 provider 结果不自动重试，转人工处理。
- 自动回复需要服务端功能开关、IM 总安全阀、用户日程开关和来源开关四层同时授权。
- 第一版只允许 Telegram Business 私聊；Telegram 群/频道/普通账号和企业微信存档不自动外发。
- pi 分析完成且确定性风险和内容检查通过后才发送。分析失败或任何未知值都 fail closed。
- adapter 回执必须区分真实发送与 dry-run；只有真实成功才能写 outgoing Message 和 `REPLIED`。

## 备选方案

- **继续由机会状态直接触发发送**：改动小，但不能表达授权、分析完成、风险和投递不确定性。
- **把 send 工具授予 pi Agent**：流程自然，但模型会同时控制判断和不可逆动作，提示词注入影响面过大。
- **依赖 Celery 重试获得可靠发送**：可提高成功率，但 Telegram 外发缺少端到端幂等键，可能重复联系客户。
- **完全禁用自动回复，只保留草稿**：风险最低，但无法满足明确的非工作时间安全接待目标。

## 后果

正面影响：发送边界可测试、可审计、来源默认关闭；分析与发送顺序稳定；dry-run 不再污染业务状态；
重复任务和人工竞态有明确处理。

成本与约束：增加迁移、账本和来源设置；部分 provider 不支持真正 exactly-once，因此不确定发送需要
人工核对；第一版可自动处理的来源和话术范围刻意较窄。

## 验证与复审

测试覆盖策略允许/拒绝矩阵、来源越权、分析失败、危险草稿、重复任务、dry-run 和人工竞态。上线
后观察 blocked reason、sending 未决记录、provider 失败和人工接管率。只有在真实 Business 私聊灰度
证明重复发送为零且危险内容拦截稳定后，才复审是否扩展来源或次数；扩展 MTProto 发送必须另写 ADR。
