# 执行计划规范

> 状态：当前规范 · 最后核验：2026-07-10

执行计划是复杂任务的可恢复状态，不是开始前写完就冻结的方案。接手者只读计划和仓库就应知道
目标、当前进度、关键发现、决策、下一步和完成证据。

## 目录

- [活动计划](active/README.md)：正在执行、尚未满足完成定义的计划。
- [已完成计划](completed/README.md)：已交付的计划与最终验证记录。
- [计划模板](template.md)：新计划模板。
- [技术债](tech-debt.md)：已确认但不属于当前改动的系统性缺口。

## 何时必须写计划

跨会话/交接、跨多个系统边界、数据库/数据迁移、认证/外部发送/部署高风险、分阶段发布或关键
需求假设不确定时必须写。小而清晰的单模块改动使用会话内计划即可。

## 生命周期

1. 复制模板为 `active/YYYY-MM-DD-<slug>.md`，填写目标、非目标、验收与影响面。
2. 实现过程中持续勾选进度，并追加发现、决策和验证结果；不要改写历史来隐藏转向。
3. 阻塞时记录已尝试证据、准确阻塞条件和需要的决定。
4. 完成后补结果总结与剩余风险，将文件移动到 `completed/`。
5. 未完成项若不再属于该计划，转入新计划或 `tech-debt.md`，不得静默丢失。

计划不替代权威文档：完成后，当前事实必须回写架构/功能/运维文档。
