# AI 开发工作流

> 状态：当前规范 · 最后核验：2026-07-10

## 1. 建立任务契约

在编辑前用可验证语言确认：目标用户、当前行为、期望行为、不做什么、验收标准。搜索相关页面、
route、use case、model、migration 和测试；不要仅凭文件名或文档推断实现状态。

根据影响面加载文档：跨层读架构，用户行为读功能地图，高风险读安全基线，运行/配置变更读运维。

## 2. 选择计划粒度

以下任一情况使用版本化执行计划：

- 预计跨越多个会话或需要交接；
- 同时修改前端、API、持久化或 worker 中两个以上边界；
- 包含 schema/data migration、外部 API、认证、真实消息发送或部署变更；
- 需求存在需要验证的关键假设；
- 有多阶段上线、兼容或回滚要求。

从 `docs/plans/template.md` 复制到 `docs/plans/active/YYYY-MM-DD-<slug>.md`。其余任务使用会话内
轻量计划，但仍要先明确验证命令。

## 3. 纵向切片实现

优先完成最小的端到端切片，而不是分别堆完所有层：

1. 先添加能表达验收标准的测试或可复现实验。
2. 从领域/契约开始明确数据和不变量。
3. 实现持久化与用例，再连接 API/worker。
4. 前端通过 `lib/api.ts` 接入真实契约，补全各异步状态。
5. 删除被替代的 mock、timer、兼容分支与死代码。

每个小步运行最窄检查。若实际发现推翻计划假设，先更新活动计划的发现/决策日志再改变方向。

## 4. 验证

验证顺序从快到慢：

1. `make harness-check`（所有改动都运行）。
2. 目标单测或单文件 lint。
3. 受影响子系统的 `make backend-check` / `make frontend-check`。
4. 跨栈、发布或高风险变更运行 `make check`，并按需做容器/API/浏览器冒烟。

测试应覆盖成功、失败、边界和权限路径。外部服务无法真实调用时说明替代验证与剩余风险，不得用
“代码看起来正确”代替证据。

## 5. 知识回写与交付

- 架构/数据流改变：更新 `docs/architecture/overview.md`。
- 用户能力或成熟度改变：更新 `docs/product/feature-map.md`。
- 命令、依赖、环境变量、队列或部署改变：更新 commands/runtime。
- 出现持久技术取舍：新增 ADR；出现无法顺手解决的真实缺口：更新技术债。
- 复杂任务完成：勾完验收与验证，把计划从 `active/` 移到 `completed/`。

交付说明必须包含：实现结果、关键文件、实际运行的检查及结果、未运行项、仍存在的风险/后续动作。

## 缺陷修复附加规则

- 先稳定复现，追到最早出现错误状态的位置，再修根因。
- 增加在修复前失败、修复后通过的回归测试。
- 检查相同模式是否存在于相邻路径，但不要扩大为无关重构。
- 若缺陷来自文档误导、缺少边界或反馈过晚，同时修复 harness 对应层。

## Review 清单

- 行为是否满足明确验收标准，而不只是“有代码”？
- 是否保持 owner 隔离、状态机、幂等和发送安全阀？
- API/类型/迁移/mapper/UI 是否完整同步？
- 失败、重试、并发和空状态是否合理？
- 测试是否会因真实回归失败，而不是只复述实现？
- 是否引入静默 mock、秘密泄露、无界查询或难以回滚的变更？
- 文档和功能成熟度是否仍准确？
