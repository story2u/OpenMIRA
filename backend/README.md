# Opportunity IM Assistant Backend

FastAPI backend for Telegram and WeCom opportunity detection, human review, and after-hours AI replies.

Python and dependencies are managed by [uv](https://docs.astral.sh/uv/). For local development,
run `uv sync --locked --dev`; add or remove dependencies with `uv add` / `uv remove` and commit both
`pyproject.toml` and `uv.lock`.

The message post-processing agent uses `@earendil-works/pi-agent-core` in
`pi-agent-runtime/`. Node dependencies are locked separately with npm because the runner is bundled
into the backend image but executed as an isolated subprocess by the Celery worker.

## Run locally with Docker

```bash
cd backend
cp .env.example .env
docker compose build
docker compose up -d postgres redis
docker compose run --rm migrate
docker compose run --rm api python scripts/seed_demo.py
docker compose up api celery_worker celery_beat
```

API docs: <http://localhost:8000/docs>

Admin-only maintenance endpoints still accept the legacy service token:

```http
Authorization: Bearer change-me
```

The frontend uses Google OAuth or Apple OAuth. On first successful OAuth login the backend creates
the user automatically and links the OAuth provider account. Configure at least one provider before
opening `/login`:

```env
GOOGLE_OAUTH_CLIENT_ID=...
GOOGLE_OAUTH_CLIENT_SECRET=...
GOOGLE_OAUTH_REDIRECT_URI=https://api.example.com/api/v1/auth/oauth/google/callback
APPLE_OAUTH_CLIENT_ID=...
APPLE_OAUTH_REDIRECT_URI=https://api.example.com/api/v1/auth/oauth/apple/callback
APPLE_OAUTH_CLIENT_SECRET=...
```

For Apple, `APPLE_OAUTH_CLIENT_SECRET` can be provided directly, or generated from
`APPLE_OAUTH_TEAM_ID`, `APPLE_OAUTH_KEY_ID`, and `APPLE_OAUTH_PRIVATE_KEY`.

When deploying through GitHub Actions, put non-sensitive values in repository or environment
variables and secrets in GitHub Secrets. The deploy workflow merges these values into the VPS
`.env` on every deployment.

`FRONTEND_BASE_URL` and `CORS_ORIGINS` are written by the deploy workflow defaults.
`JWT_SECRET_KEY` is generated once on the VPS if missing and then preserved in `.env`.

GitHub Variables:

```bash
gh variable set GOOGLE_OAUTH_CLIENT_ID --body "..."
gh variable set GOOGLE_OAUTH_REDIRECT_URI --body "https://im.story2u.xyz/api/v1/auth/oauth/google/callback"
gh variable set APPLE_OAUTH_CLIENT_ID --body "..."
gh variable set APPLE_OAUTH_REDIRECT_URI --body "https://im.story2u.xyz/api/v1/auth/oauth/apple/callback"
gh variable set PI_AGENT_ENABLED --body "true"
gh variable set PI_AGENT_PROVIDER --body "deepseek"
gh variable set PI_AGENT_MODEL --body "deepseek-v4-flash"
```

GitHub Secrets:

```bash
gh secret set GOOGLE_OAUTH_CLIENT_SECRET --body "..."
gh secret set APPLE_OAUTH_CLIENT_SECRET --body "..."
gh secret set CLOUDFLARE_TUNNEL_TOKEN --body "..."
# The deploy workflow maps this provider-specific secret to PI_AGENT_API_KEY.
gh secret set DEEPSEEK_API_KEY --body "..."
```

The pinned pi 0.80.6 catalog exposes `deepseek-v4-flash` and `deepseek-v4-pro`. Use `flash` for the
default message-processing path; switch the `PI_AGENT_MODEL` variable to `deepseek-v4-pro` only when
the additional reasoning quality justifies its latency and cost. When `PI_AGENT_PROVIDER=deepseek`,
the deploy workflow prefers `DEEPSEEK_API_KEY` and falls back to a generic `PI_AGENT_API_KEY` secret.

If Apple client secret should be generated by the backend instead, set `APPLE_OAUTH_TEAM_ID` and
`APPLE_OAUTH_KEY_ID` as variables, then set `APPLE_OAUTH_PRIVATE_KEY` as a secret.

## Telegram user-client ingestion

Telegram Bot API cannot read a group unless the bot is added to that group. For job groups where
you are only a normal member, use the MTProto user-client listener. It logs in with your own
Telegram account session, reads only chats that account can already access, and feeds messages into
the same opportunity detection pipeline as the webhook adapters.

1. Create a Telegram API app at <https://my.telegram.org/apps> and copy `api_id` and `api_hash`.
2. Log in with Google or Apple and open `/settings/telegram`.
3. Enter `api_id`, `api_hash`, and your Telegram phone number.
4. Send and verify the login code. If the account has two-step verification, enter that password.
5. Load dialogs, add the target job group or channel, enable ingestion, and save.

The encrypted `api_hash` and session string are stored in PostgreSQL per user. The
`telegram_listener` service polls enabled database monitor rows and starts one listener per monitor.
The monitor limit comes from the authenticated user's versioned subscription entitlement: Free
allows one Telegram monitor, while Plus/Pro/Max share a 10/50/100-group combined Telegram + WeCom
limit. It only ingests messages by default. It does not send AI replies from your personal Telegram
account. After a downgrade, excess Telegram monitors are quota-paused rather than deleted; the user
chooses which monitor IDs to retain through the retention endpoint or `/settings/telegram`.
